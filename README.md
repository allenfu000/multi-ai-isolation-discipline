# 双 AI 物理隔离协作的工程纪律

## 适用人群

- 同时使用两个或多个 AI 产品(Claude / ChatGPT / Codex 等)
- 想用双 AI 对照减少模型偏差

## 核心原则

**两个/多个 AI 之间的全电脑信息流必须由人脑严格拆开,AI 之间不能有任何直接或间接的内容共享。**

如果违反这条原则,则所有"双/多 AI 协作"都会最终趋同,失去对照价值。

(比如数据接口采集这种工作,必须控制不同 AI 先后分别采集数据,哪怕相同数据,接受时间成本。)

---

## 一、隔离

- **两个独立根目录**:`claude-projects\` 和 `codex-projects\`,每个项目永远归属一个 AI,永不跨根目录
- **桌面应用图标、终端、快捷方式按物理上的视觉分离**,明确意识到自己进入了哪个 AI 的工作流
- **不建任何共享文件夹,包括只读共享**
- **项目不跨 AI 接手,换 AI 必须重做方案!** 接手的 AI 一旦读了原方案就必被锁死,后续只是在原框架内做局部修改,无法产生独立视角、独立特点
- **方案的源头永远是单 AI 产品线闭环,并做双线**——启动 Codex 项目就重新和 ChatGPT 网页版聊一份独立方案,启动 Claude Code 项目就重新和 Claude 网页版聊一份独立方案,**两个 AI 之间永远没有直接通信**

---

## 二、审计流程

| 步骤 | 操作 | 关键纪律 |
|---|---|---|
| 1 | Claude 和 Codex 各自在隔离环境下完工(完整完工,不是中途) | 半成品不能审计,半成品的"问题"分不清是 bug 还是没做完 |
| 2 | 把 Codex 的产物**整份复制到一个全新的、独立的桌面文件夹** | 此前没装过任何东西的干净文件夹 |
| 3 | 在新文件夹里启动**一个新的 Claude 窗口**(不是原 Claude 窗口) | 新窗口意味着没有原 Claude 的工作上下文,纯粹做翻译 |
| 4 | 让新 Claude 窗口**把 Codex 产物翻译成 Claude 风格的版本——纯翻译机角色,只改风格,不做任何判断** | 严格保持原工程和代码逻辑,只改目录、命名、库选择等非工程根基风格;遇到它认为有问题的地方**全部忽略**,只做"Claude 风格版" |
| 5 | 让**原 Claude 窗口**对照"翻译版"和自己的版本,指出底层逻辑差异 | 原 Claude 窗口是唯一的审计者,新 Claude 窗口绝不能介入审计 |
| 6 | 人脑作为**副审计**:补充原窗口可能没指出的内容,然后决定吸收哪些建议、怎么改 | 人脑不是被动接受 AI 对照结果,而是参考它+查漏补缺 |

---

**为什么必须翻译**:

直接对比原始 Codex 产物 ↔ 原始 Claude 产物,差异里混了两种东西:

- **风格差异**(命名习惯、结构偏好、库选择)——这是噪音
- **底层逻辑差异**(对同一问题的真实判断分歧)——这是信号

**不做翻译就变成寻找对方噪音**——寻找出来的"问题"绝大部分是风格习惯不同,不是真分歧,翻译之后两份产物风格统一,**剩下的差异才是值得深究的真问题**。

---

**新 Claude 窗口必须是纯翻译机,不能标注问题**:

新 Claude 窗口在翻译过程中,默认倾向就是修正它认为 Codex 写错的地方。一旦它开始做判断,**最终审计权就被分裂在两个 Claude 窗口之间,对照机制失效**。

更严重的情况:如果新 Claude 窗口直接进行工程修改,那这份产物就是 Claude 的接手版,不再是 Codex 逻辑的载体,**项目实质上已经合并了,双 AI 对照彻底失效**。

权责必须清晰:

- **新 Claude 窗口 = 纯翻译机**(只改风格,完全忽略它认为的问题)
- **原 Claude 窗口 = 唯一审计者**(看统一风格后的两份产物,指出底层逻辑分歧)
- **人脑 = 副审计**(参考原窗口的对照结果+补充它没指出的内容+做最终决策)

## 附注

**非程序员慎用执行力强的 Codex**,指令有歧义时它会陪你套娃,人机一起在错误链路上高效循环,非程序员很难自己察觉,因为它以执行能力强并以范围内严谨落地为首要目标。(非程序员开始严谨套娃)

**使用批判版 Claude 时要守住主线**,逻辑不可能"完美",容易无限指出问题、无限改主线则项目永不完工,因为它协作能力强、会持续陪你深挖并以全局内清晰执行为首要目标。(非程序员开始套娃修复)
